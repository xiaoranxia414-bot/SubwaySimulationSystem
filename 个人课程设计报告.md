## 一、选题背景与研究意义

城市轨道交通作为大城市公共交通体系的核心组成部分，其车站内部的客流组织效率直接关系到运营安全与乘客出行体验。在高峰时段，售票、安检、闸机及换乘通道等区域极易形成拥堵瓶颈，严重时可能引发安全隐患。因此，在系统设计阶段借助仿真手段预测客流规律、识别拥堵节点、优化设施配置，已成为地铁运营规划的重要研究方向。

本课程设计旨在构建一个高保真、可扩展的地铁站人群流动仿真系统。该系统以数据结构与算法为核心支撑，综合运用图论建模、有限状态机、空间哈希、多目标优化等技术，对地铁站内人群的生成、移动、排队、上下车等完整行为进行离散时间步仿真，并辅以可视化界面与数据分析模块，为运营决策提供定量依据。

## 二、系统总体架构设计

本系统采用分层模块化架构，自上而下分为数据模型层、业务逻辑层与展示层三个层次。

数据模型层负责地铁站拓扑结构、乘客对象及列车对象的抽象与存储。其中，地铁站被抽象为基于有向图的拓扑模型，乘客对象基于有限状态机刻画其完整生命周期，列车对象则负责模拟线路上的车辆运行规律。

业务逻辑层包含仿真引擎、路径规划器与数据分析模块。仿真引擎以离散时间步驱动整个系统的运行，处理乘客生成、状态转移、队列服务、冲突检测及密度更新等核心逻辑；路径规划器为每位乘客提供符合其偏好策略的行进路径；数据分析模块则在仿真过程中持续采集各项指标，并在仿真结束后生成统计报告。

展示层基于 PyQt5 构建，提供拓扑图编辑、热力图渲染、实时数据监控、分析报告展示及轨迹回放等功能，支持用户对仿真过程进行交互式观察与参数调整。

该架构的确立经历了数次迭代。初期曾考虑将逻辑层与展示层耦合于同一模块，导致代码可读性与可维护性显著下降。经重构后，各层职责边界清晰，模块间通过明确接口进行通信，符合软件工程中的单一职责原则与开闭原则。

## 三、数据结构与算法设计

### 3.1 地铁站拓扑图 —— 有向图（DiGraph）

#### 3.1.1 数据结构选型论证

地铁站由多个功能区域及其连接通道构成，其结构天然符合图模型。考虑到乘客流动具有显著的方向性，且系统需要支持最短路径、多目标优化等图算法，本系统选用 NetworkX 库的 DiGraph（有向图）作为底层存储结构，并在此基础上封装 StationGraph 类。

在数据结构选型阶段，曾对三种候选结构进行全面评估：

| 候选结构 | 空间复杂度 | 边查询 | 邻接遍历 | 路径规划支持 | 结论 |
|---------|-----------|--------|---------|------------|------|
| 邻接矩阵 | O(V²) | O(1) | O(V) | 需手写算法 | 稀疏图空间浪费严重，不适用 |
| 自定义邻接表 | O(V+E) | O(deg(v)) | O(deg(v)) | 需手写算法 | 空间最优，但开发成本高、易出错 |
| NetworkX DiGraph | O(V+E) | O(deg(v)) | O(deg(v)) | 内置 Dijkstra/BFS/DFS | 空间效率与开发效率兼顾，选定 |

对于地铁站这类稀疏图，节点数 V 与边数 E 满足 E ≪ V²。以中型站为例，V ≈ 17，E ≈ 32，邻接矩阵需分配 289 个存储单元，其中有效项仅占 11%，内存浪费率高达 89%。自定义邻接表虽在空间效率上达到理论最优，但需自行实现图遍历、最短路径、连通性检测等算法，开发周期与调试成本难以接受。NetworkX DiGraph 底层采用优化的邻接表字典结构，同时内置了丰富的图算法接口与可视化支持，在开发效率与运行性能之间取得了良好平衡。

StationGraph 的核心结构定义如下：

```python
class StationGraph:
    def __init__(self):
        self.graph = nx.DiGraph()      # 底层有向图
        self._type_index = {}           # 按类型索引（缓存）
        self._floor_index = {}          # 按楼层索引（缓存）
        self._dirty = True              # 索引脏标记
```

#### 3.1.2 惰性索引机制

直接使用 DiGraph 进行类型查询时，每次调用 `get_nodes_by_type()` 均需遍历全部节点，时间复杂度为 O(V)。在仿真运行阶段，该操作每步执行多次（如查找所有入口以生成新乘客、查找所有站台以统计候车人数），成为显著的性能瓶颈。

为解决该问题，本系统引入了惰性索引机制。StationGraph 内部维护两个字典：`_type_index` 按节点类型映射到节点列表，`_floor_index` 按楼层映射到节点列表。当图结构发生变更（如添加、删除节点）时，设置 `_dirty = True`；查询操作前检查该标志，若已置位则触发索引重建，重建完成后清除脏标记。

索引重建的伪代码如下：

```
算法: RebuildIndex
输入: 图 G
输出: 更新后的 _type_index 与 _floor_index

1. _type_index ← 空字典
2. _floor_index ← 空字典
3. for each (node, data) in G.nodes(data=True):
4.   _type_index[data['type']].append(node)
5.   _floor_index[data.get('floor', 0)].append(node)
6. _dirty ← False
```

单次索引重建的时间复杂度为 O(V)，但该操作仅在图结构变更时触发。在仿真运行阶段，图结构保持不变，查询操作可直接命中缓存索引，时间复杂度降至 O(1)。设图结构变更次数为 M，总查询次数为 Q，则均摊时间复杂度为 O(M·V + Q)，当 Q ≫ M 时，均摊查询复杂度趋近于 O(1)。

边属性（edge_attributes）设计如下：

#### 3.1.4 多楼层支持

本系统支持多楼层拓扑建模，楼层属性 floor 为整数值：0 表示地面层，1 表示站厅层，-1 表示地下一层，依此类推。系统提供 `get_nodes_by_floor(floor)` 接口，返回指定楼层的全部节点，该接口基于惰性索引实现，查询复杂度为 O(1)（均摊）。

默认拓扑按楼层分层布局：地面层布置入口与出口，站厅层布置售票、安检、闸机与通道，站台层布置楼梯、扶梯与站台。节点在可视化界面中的边框样式按楼层区分：地下层采用蓝色虚线，地面层采用黑色实线，站厅层采用橙色实线，更高层采用紫色实线，从而在二维视图中直观表达三维空间结构。

### 3.2 乘客对象 —— 有限状态机（FSM）

#### 3.2.1 FSM 选型论证

乘客从进站到出站需经历多个离散阶段，各阶段的行为规则差异显著。若采用传统的嵌套条件语句实现，代码将呈现高度耦合的复杂结构，状态转移逻辑分散于多个函数中，难以维护与扩展。有限状态机（Finite State Machine, FSM）将复杂行为拆解为确定性的状态转移，每个状态对应独立的行为处理器，新增状态仅需扩展处理器字典，无需修改现有代码，符合开闭原则。

#### 3.2.2 状态定义与转移图

乘客生命周期被定义为九个状态：

| 状态常量 | 值 | 含义 |
|---------|-----|------|
| STATE_ENTERING | 'entering' | 刚进入车站 |
| STATE_TICKET | 'ticket' | 购票/刷卡（需排队服务） |
| STATE_SECURITY | 'security' | 安检（需排队服务） |
| STATE_GATE | 'gate' | 过闸机（需排队服务） |
| STATE_WALKING | 'walking' | 在通道中行走 |
| STATE_WAITING | 'waiting' | 站台候车 |
| STATE_BOARDING | 'boarding' | 上车过程 |
| STATE_RIDING | 'riding' | 乘车中 |
| STATE_EXITING | 'exiting' | 出站 |

状态转移关系如下：

```
entering → walking → [ticket] → walking → security → walking → gate → walking
                    ↑ (可选跳过)                                      ↓
platform ← stairs/escalator ← corridor ←──────────────────────────────┘
   ↓
waiting → boarding → riding → walking → exiting → [移除]
```

其中 ticket 状态为可选：无票乘客到达 ticket 节点时，依据其熟悉度 familiarity 以概率 `skip_prob = 0.4 + familiarity × 0.4` 直接跳过购票环节，进入 walking 状态继续向闸机移动。该概率设计基于如下假设：熟悉度为 0 的乘客（首次乘车）有 40% 概率跳过售票，熟悉度为 1 的乘客（常客）有 80% 概率跳过，反映出现代地铁场景中移动支付与交通卡普及后的实际通行模式。

#### 3.2.3 状态机调度实现

状态调度采用 Python 字典分发（dict dispatch）实现：

```python
def update_state(self, simulation):
    handlers = {
        self.STATE_ENTERING:  self._handle_entering,
        self.STATE_TICKET:    self._handle_ticket,
        self.STATE_SECURITY:  self._handle_security,
        self.STATE_GATE:      self._handle_gate,
        self.STATE_WALKING:   self._handle_walking,
        self.STATE_WAITING:   self._handle_waiting,
        self.STATE_BOARDING:  self._handle_boarding,
        self.STATE_RIDING:    self._handle_riding,
        self.STATE_EXITING:   self._handle_exiting,
    }
    handler = handlers.get(self.current_state)
    if handler:
        handler(simulation)
```

该实现的时间复杂度为 O(1)，相较于冗长的 if-else 链具有更高的执行效率与可读性。

#### 3.2.4 乘客属性设计

| 属性 | 分布类型 | 参数 | 含义 |
|------|---------|------|------|
| speed | 正态分布 | N(1.2, 0.2) | 行走速度（m/s） |
| patience | Beta 分布 | α=2, β=2 | 耐心度，影响重规划阈值 |
| familiarity | Beta 分布 | α=2, β=2 | 熟悉度，影响服务时长与跳过售票概率 |
| has_ticket | 伯努利分布 | p=0.5 | 是否有票 |

Beta 分布的选取理由在于其值域天然约束于 [0, 1]，且通过调整 α、β 参数可灵活控制分布形态。本系统取 α=β=2，使大部分乘客的耐心度与熟悉度集中在中等水平，少数偏高或偏低，符合实际人群特征。

### 3.3 空间哈希（SpatialHash）

#### 3.3.1 数据结构选型论证

碰撞检测是人群仿真中的核心操作，用于判断两名乘客是否占据同一空间位置。若采用暴力两两比较，对于 N 名乘客需执行 N(N-1)/2 次距离计算，时间复杂度为 O(N²)。当 N=2000 时，每步需进行约 200 万次比较，计算开销不可接受。

空间哈希（Spatial Hashing）将二维空间划分为固定大小的网格单元，仅检查同一单元及相邻单元（共 9 个单元）内的实体对，将平均比较次数从 O(N²) 降至 O(N)。其原理基于如下观察：若两名实体的距离小于碰撞半径 r，则它们必定位于同一网格单元或相邻网格单元内，因此无需检查距离较远的单元。

#### 3.3.2 实现原理

SpatialHash 类的核心结构为字典 `grid: dict[(int, int), list]`，键为单元坐标 (cell_x, cell_y)，值为该单元内的实体列表。单元坐标通过下取整除法计算：

```python
def _get_cell_key(self, x, y):
    return (int(x // self.cell_size), int(y // self.cell_size))
```

插入、删除与更新操作的时间复杂度均为 O(1)。邻居查询时，先定位目标实体所在的单元，再遍历该单元及其 8 个相邻单元内的全部实体，返回潜在碰撞对。

#### 3.3.3 自适应单元格策略

单元格大小直接影响查询效率与内存占用。单元格过大时，单个单元内实体数增多，邻居查询退化为近似线性扫描；单元格过小时，实体跨单元频繁，需检查的相邻单元数增多。本系统采用自适应策略，根据实体数量动态调整 cell_size：

| 实体数量 | 单元格大小 |
|---------|-----------|
| < 100 | 2.0 m |
| 100 ~ 1000 | 1.5 m |
| > 1000 | 1.0 m |

#### 3.3.4 复杂度分析

| 操作 | 时间复杂度 | 空间复杂度 |
|------|-----------|-----------|
| 插入 | O(1) | O(1) |
| 删除 | O(1) | O(1) |
| 更新 | O(1) | O(1) |
| 邻居查询 | O(k)，k 为邻近单元实体数 | O(N) |

在均匀分布假设下，k ≈ N · (2r/cell_size)²，其中 r 为碰撞半径。当 cell_size 取 2r 时，k ≈ N/9，即每个单元平均包含 N/9 个实体，邻居查询需检查约 N 个实体，总体复杂度为 O(N)。

### 3.4 优先队列（PriorityQueue）

路径规划中的 Dijkstra 与 A* 算法需要频繁执行"取最小值"操作。本系统采用 Python 标准库 heapq 实现的二叉堆优先队列：

| 操作 | 时间复杂度 | 说明 |
|------|-----------|------|
| push(item, priority) | O(log N) | 插入元素并维护堆性质 |
| pop() | O(log N) | 取出优先级最小元素并调整堆 |
| peek() | O(1) | 查看最小元素，不删除 |

相较于线性扫描的 O(N)，二叉堆在图规模较大时优势显著。以 50 节点图为例，Dijkstra 算法需执行约 V 次 pop 与 E 次 push，总时间复杂度为 O((V+E) log V)。

## 四、核心算法设计

### 4.1 路径规划算法

#### 4.1.1 算法需求分析

本系统需提供至少四种单目标路径模式及一种多目标综合优化模式。各模式的权重函数与适用场景如下：

| 模式 | 权重函数 | 算法 | 适用场景 |
|------|---------|------|---------|
| shortest_distance | w = distance | Dijkstra | 距离敏感型乘客 |
| shortest_time | w = base_time × congestion + distance/speed | Dijkstra | 时间敏感型乘客 |
| min_switches | w = distance + 20×switch_penalty | 改进 Dijkstra | 换乘厌恶型乘客 |
| least_congested | w = congestion_factor | Dijkstra | 舒适度优先型乘客 |
| multi_objective | Pareto 前沿 + TOPSIS | 多目标 A* | 综合最优（默认） |

#### 4.1.2 Dijkstra 算法实现

最短路径算法采用经典 Dijkstra 算法，基于二叉堆优先队列实现：

```
算法: Dijkstra(G, s, t, weight)
输入: 图 G, 起点 s, 终点 t, 权重函数 weight
输出: 最短路径节点列表

1. 初始化: dist[v] ← ∞ (∀v∈V), dist[s] ← 0, prev[v] ← null (∀v∈V)
2. 优先队列 Q ← {(0, s)}
3. while Q 非空:
4.   (d, u) ← Q.pop()
5.   if u == t: break
6.   if d > dist[u]: continue  // 处理过期条目
7.   for each v ∈ G.neighbors(u):
8.     w ← weight(u, v, G[u][v])
9.     if dist[u] + w < dist[v]:
10.      dist[v] ← dist[u] + w
11.      prev[v] ← u
12.      Q.push((dist[v], v))
13. 回溯 prev[] 数组，从 t 到 s 构建路径并反转
```

**时间复杂度**：O((V + E) log V)（二叉堆实现）  
**空间复杂度**：O(V)

对于 shortest_time 模式，权重函数综合考虑了基础通行时间、拥堵系数与距离：

```
weight(u, v) = base_time(u,v) × congestion_factor(u,v) + distance(u,v) / speed
```

其中 congestion_factor 为动态变量，随仿真过程实时更新，因此该模式下的路径会随拥堵状况变化而调整。

### 4.2 多目标优化算法 —— Pareto 前沿 + TOPSIS

#### 4.2.1 问题建模

单一目标（如最短时间）往往导致其他指标恶化（如距离过长、换乘过多）。本系统的 multi_objective 模式同时优化四个维度：时间最短、距离最短、区域切换最少、拥堵度最小。

定义四维代价向量：

```
c(path) = (time(path), distance(path), switches(path), congestion(path))
```

#### 4.2.2 Pareto 支配关系

对于两条路径 p₁ 与 p₂，称 p₁ 支配 p₂，当且仅当：

```
∀i ∈ {1,2,3,4}: c_i(p₁) ≤ c_i(p₂)  ∧  ∃j ∈ {1,2,3,4}: c_j(p₁) < c_j(p₂)
```

Pareto 前沿定义为所有不被任何其他路径支配的路径集合：

```
ParetoFront = {p | ∄p' ≠ p, p' 支配 p}
```

#### 4.2.3 Pareto 前沿搜索算法

采用改进的多目标 A* 搜索：

```
算法: ParetoSearch(G, s, t, max_paths)
输入: 图 G, 起点 s, 终点 t, 最大路径数上限
输出: Pareto 前沿路径集合

1. pareto_by_node ← {s: [(0,0,0,0)]}
2. Q ← [(0,0,0,0, s, [s])]
3. found_paths ← []
4. while Q 非空且 |found_paths| < max_paths:
5.   (time, dist, switch, congest, u, path) ← Q.pop()
6.   if u == t:
7.     found_paths.append((path, (time, dist, switch, congest)))
8.     continue
9.   for each v ∈ G.neighbors(u):
10.    new_cost ← 累加(u,v)边的各项代价
11.    if new_cost 不被 pareto_by_node[v] 中任何解支配:
12.      从 pareto_by_node[v] 移除被 new_cost 支配的旧解
13.      pareto_by_node[v].append(new_cost)
14.      Q.push((new_cost..., v, path+[v]))
```

支配检查的剪枝效果使得搜索空间大幅缩小。实验表明，50 节点图中 Pareto 前沿通常包含 3~8 条路径，搜索时间约为 5~20 毫秒。

#### 4.2.4 TOPSIS 折中选择

Pareto 前沿给出了一组互不支配的候选解，需从中选择最符合用户偏好的折中方案。本系统采用 TOPSIS（逼近理想解排序法），其算法步骤如下：

**Step 1: 构建决策矩阵**

设 Pareto 前沿包含 m 条路径，4 个优化目标，则决策矩阵 X 为 m×4 矩阵：

```
X = [x_ij],  i = 1,...,m,  j = 1,...,4
```

其中 x_ij 为第 i 条路径在第 j 个目标上的代价值。

**Step 2: 归一化**

```
r_ij = x_ij / √(Σ_{k=1}^{m} x_kj²)
```

**Step 3: 加权**

本系统采用如下权重分配：时间 0.40、距离 0.25、切换次数 0.20、拥堵度 0.15。

```
v_ij = r_ij × w_j
```

**Step 4: 确定理想解与负理想解**

由于所有目标均为越小越好：

```
V⁺ = (min(v_·1), min(v_·2), min(v_·3), min(v_·4))   // 理想解
V⁻ = (max(v_·1), max(v_·2), max(v_·3), max(v_·4))   // 负理想解
```

**Step 5: 计算欧氏距离**

```
D⁺_i = √Σ_{j=1}^{4} (v_ij - V⁺_j)²
D⁻_i = √Σ_{j=1}^{4} (v_ij - V⁻_j)²
```

**Step 6: 计算贴近度并排序**

```
C_i = D⁻_i / (D⁺_i + D⁻_i)
```

C_i ∈ [0, 1]，值越大表示该路径越接近理想解。返回 C_i 最大的路径作为最优推荐。

**复杂度分析**：
- Pareto 搜索：最坏 O(k × (V+E) log V)，k 为前沿路径数（通常 ≤ 10）
- TOPSIS：O(m × n)，m 为候选数，n 为目标数

#### 4.2.5 路径缓存机制

同一段路径在短时间内被反复查询的概率极高。本系统引入路径缓存，缓存键为 `(start_node, end_node, mode)` 三元组：

```python
def find_path(self, start_node, end_node, mode='shortest_time'):
    cache_key = (start_node, end_node, mode)
    if cache_key in self.path_cache:
        self.cache_hits += 1
        return self.path_cache[cache_key]
    
    self.cache_misses += 1
    # 执行路径搜索...
    if path:
        self.path_cache[cache_key] = path
    return path
```

缓存清理策略：当检测到某条边的拥堵系数超过阈值（默认 0.5）时，清空 `shortest_time` 与 `multi_objective` 模式的缓存条目，因为这两种模式对拥堵变化最为敏感。实验表明，重复查询场景下缓存命中率可达 95% 以上。

### 4.3 队列服务算法

地铁站内的服务节点遵循先到先服务（FCFS）原则。本系统采用 Python collections.deque 实现真实的 FIFO 队列，每步根据泊松分布抽样确定服务人数：

```
算法: ProcessQueues(queues, service_rates)
1. for each service_node in queues:
2.   queue ← queues[node_id]
3.   rate ← service_rates[node_type]          // 人/秒
4.   capacity ← max(1, Poisson(rate))         // 泊松抽样
5.   served ← 0
6.   while queue 非空且 served < capacity:
7.     passenger ← queue[0]
8.     if passenger 仍在该节点且处于服务状态:
9.       queue.popleft()
10.      passenger.current_state ← STATE_WALKING
11.      served += 1
12.    else:
13.      queue.popleft()                        // 清理已离开乘客
```

服务完成后，乘客状态由服务状态切换至 STATE_WALKING，再由 move() 方法驱动其向下一节点移动，而非直接瞬移，从而保证了移动过程的真实性与连续性。

### 4.4 客流生成算法

#### 4.4.1 非齐次泊松过程

地铁客流在宏观上符合泊松分布，但到达率随时间显著变化。本系统采用非齐次泊松过程，到达率 λ(t) 为时间 t 的函数：

```
λ(t) = λ_base + Σ peak_i(t)
```

其中 peak_i(t) 为高斯型时段峰值函数：

```
peak_i(t) = A_i × exp(-((t - μ_i) / σ_i)²)
```

系统内置两个典型时段：
- 早高峰：μ = 08:00，σ = 1h，峰值倍率 A = 3.0×λ_base
- 晚高峰：μ = 18:00，σ = 0.8h，峰值倍率 A = 2.5×λ_base

#### 4.4.2 乘客属性采样

| 属性 | 分布 | 参数 | 值域 |
|------|------|------|------|
| 速度 | 正态分布 | N(1.2, 0.2) | (0, +∞) |
| 耐心度 | Beta 分布 | B(2, 2) | [0, 1] |
| 熟悉度 | Beta 分布 | B(2, 2) | [0, 1] |

Beta 分布的概率密度函数为：

```
f(x; α, β) = x^(α-1) (1-x)^(β-1) / B(α, β)
```

当 α = β = 2 时，分布呈单峰对称形态，峰值位于 x = 0.5，两侧逐渐衰减，符合"大多数人处于中等水平"的人群特征假设。

### 4.5 密度-速度模型

本系统采用 Weidmann 行人基本图的简化连续版本，描述行人密度 ρ 与速度 v 的关系：

```
v(ρ) = v_free × max(0.05, 1 - (ρ / ρ_max)^γ)
```

其中：
- v_free = 1.2 m/s，为自由流速度
- ρ_max = 5.0 人/m²，为最大密度
- γ = 0.3，为衰减系数

早期版本采用阶梯式分段函数，在密度阈值处产生速度突变，导致仿真画面出现明显的"集体卡顿"现象。改为连续衰减函数后，速度随密度平滑变化，视觉效果与物理合理性均得到显著改善。

### 4.6 冲突检测算法

#### 4.6.1 暴力法

冲突检测的核心任务是判断两名乘客是否处于同一空间位置。暴力法对全部乘客执行两两距离计算：

```
算法: BruteForceCollision(passengers)
1. conflicts ← []
2. for i ← 0 to N-1:
3.   for j ← i+1 to N-1:
4.     if distance(passengers[i], passengers[j]) < threshold:
5.       conflicts.append((i, j))
6. return conflicts
```

时间复杂度为 O(N²)，空间复杂度为 O(1)（不计输出）。当 N = 2000 时，每步需执行约 200 万次距离计算，计算开销不可接受。

#### 4.6.2 空间哈希优化

改用空间哈希后，冲突检测流程如下：

```
算法: SpatialHashCollision(spatial_hash, passengers)
1. spatial_hash.clear()
2. for each p in passengers:
3.   spatial_hash.insert(p)
4. conflicts ← []
5. for each p in passengers:
6.   neighbors ← spatial_hash.query_neighbors(p)
7.   for each q in neighbors:
8.     if q.id > p.id and distance(p, q) < threshold:
9.       conflicts.append((p, q))
10. return conflicts
```

时间复杂度降至 O(N)，空间复杂度为 O(N)。在 1000 人规模的测试场景中，单步冲突检测时间从约 120ms 降至约 3ms，性能提升约 40 倍。

## 五、场景设计

本系统支持小型站、中型站、大型站及换乘站四种规模场景。各场景均采用多楼层拓扑结构，节点坐标依据功能流程递进排布，同类节点沿 y 轴等距分布。具体布局如下：

| 楼层 | floor | 节点类型 |
|------|-------|---------|
| 地面层 | 0 | entrance, exit |
| 站厅层 | 1 | ticket, security, gate, corridor |
| 站台层 | -1 | stairs, escalator, platform |

节点容量、面积、基础速度等属性随站点规模递增。以中型站为例：入口容量 50 人，售票容量 30 人，安检容量 20 人，闸机容量 15 人，站台容量 200 人。通道长度依据节点坐标几何距离计算，宽度依据类型设定：主通道 4m，楼梯 2m，扶梯 3m。

## 六、前端界面设计

前端界面基于 PyQt5 框架构建，采用 QMainWindow 配合 QTabWidget 的架构。左侧为控制面板，提供仿真参数配置与运行控制；右侧为主显示区，包含拓扑图、热力图、实时数据、分析报告及轨迹回放五个标签页。

拓扑图编辑模块基于 QGraphicsScene/QGraphicsView 实现。节点采用 QGraphicsEllipseItem 子类封装，支持拖拽、选中、属性编辑；边采用 QGraphicsLineItem 子类封装，拖拽时通过 scenePos() 获取全局坐标以自动更新位置。节点拥堵状态通过颜色渐变呈现：基础颜色随类型固定，拥堵时红色分量递增、绿色分量递减，形成直观的视觉警示。多楼层可视化通过边框样式区分：地下层（floor < 0）采用蓝色虚线，地面层（floor = 0）采用黑色实线，站厅层（floor = 1）采用橙色实线，更高层采用紫色实线。

实时数据折线图基于 Matplotlib FigureCanvas 嵌入 Qt，展示乘客数量、平均密度与最长队列三条曲线。交互功能包括：滚轮缩放（以鼠标位置为中心）、左键拖拽平移（X 轴边界限制于 06:00~24:00）、复位视图按钮、Y 轴自适应。热力图采用 YlOrRd 色阶渲染节点密度，辅以密度数值标注与 Colorbar。

中文显示方面，系统通过动态检测系统可用字体解决跨平台乱码问题。检测优先级为：SimHei → Microsoft YaHei → PingFang SC → WenQuanYi Zen Hei → Noto Sans CJK SC。检测逻辑基于 QFontDatabase().families() 获取系统可用字体集，选取优先级最高的可用字体作为全局字体。Matplotlib 后端配置采用相同的轮询机制，并设置 `axes.unicode_minus = False` 以修复负号显示异常。

## 七、后端开发

后端的核心为仿真引擎（SimulationEngine），其主循环 step() 按以下顺序执行：

```
1. _generate_passengers()     // 非齐次泊松过程生成新乘客
2. _process_queues()          // FIFO 队列服务
3. _update_trains()           // 列车到站/离站/上下车
4. _update_passengers()       // 状态机转移 + 移动
5. _handle_conflicts()        // 空间哈希冲突检测
6. _update_densities()        // 密度计算与拥堵系数更新
7. _collect_stats()           // 统计数据采集
```

列车系统（Train）具备发车间隔、停站时间、容量上限及车门状态控制等属性。列车到站后开门，候车乘客按 FIFO 顺序上车；容量达到上限后，剩余乘客继续等待后续列车。该机制使站台候车行为的仿真更加贴近实际运营规律。

仿真引擎采用三阶段遍历策略：首先遍历全部乘客更新状态与位置，然后处理冲突与密度，最后采集统计。该策略避免了同一阶段内乘客状态互相干扰的问题，确保了每步内部的一致性。

## 八、轨迹回放功能

为支持仿真结果的回溯分析，本系统开发了轨迹回放功能。仿真线程在运行过程中，每步记录一次状态快照，数据结构如下：

```python
snapshot = {
    'step': int,                // 当前仿真步数
    'passenger_count': int,     // 站内总人数
    'densities': dict,          // 节点 → 实时密度
    'node_counts': dict,        // 节点 → 当前人数
}
```

轨迹回放视图（TrajectoryPlaybackView）复用 TopologyView 的显示逻辑，禁用编辑与拖拽功能。界面提供 QSlider 时间轴滑块，支持 0~N 步的任意定位；配备播放/暂停按钮及 QTimer 控制的自动回放，速度倍率可选 0.5x、1x、2x、5x、10x。回放过程中，节点人数标签与拥堵颜色依据历史数据动态更新，边的拥堵粗细同步变化，完整还原人流演化过程。

轨迹回放功能的实现将数据记录与可视化展示拆分为两个独立环节：仿真线程在运行过程中持续记录状态快照，仿真结束后将完整的历史数据列表传递给回放视图；回放视图则专注于按时间顺序读取这些数据并驱动界面刷新。这种拆分避免了仿真逻辑与回放逻辑的相互耦合，使两个模块可以独立开发与维护。

### 9.2 集成测试与关键 Bug 修复

**Bug 1：乘客堆积于入口**

现象：仿真运行后全部乘客堆积在 entrance1 节点，系统假死。

根因分析：`_can_move_to()` 方法使用硬编码的 `node['capacity']` 作为移动门槛，而服务节点容量设置过小（如 gate 仅 10 人），边 capacity 也仅 15 人，导致通道瞬间饱和。

解决方案：
- 节点移动限制改为基于面积 × 最大密度的动态密度限制（服务节点 8 人/m²，一般区域 5 人/m²）
- 边通行能力改为基于边面积/人均占用面积计算（`width × distance / 0.5`）
- 修正状态机逻辑：服务完成后转为 STATE_WALKING 以触发移动

**Bug 2：服务完成后状态死锁**

现象：乘客在 ticket/security/gate 服务完成后停留在原地不移动。

根因分析：服务完成后直接将状态设为下一个服务状态（如 ticket 完成后设为 security），而非先回到 walking 状态。由于只有 walking 状态才会触发 move()，乘客因此停留在原地。

解决方案：服务完成后统一将状态切换为 STATE_WALKING，由 move() 驱动乘客走向下一节点，抵达后根据节点类型再触发相应的服务状态。

## 十、总结

本课程设计围绕地铁站人群流动仿真这一实际应用场景，完成了从数据结构选型、核心算法设计、前后端开发到系统测试的完整开发流程。

在数据结构层面，有向图（NetworkX DiGraph）、有限状态机、空间哈希与优先队列的合理组合为系统提供了坚实的底层支撑。DiGraph 的选择基于对邻接矩阵、自定义邻接表及 NetworkX 三者的综合评估；惰性索引机制将类型查询的均摊复杂度降至 O(1)；空间哈希将冲突检测从 O(N²) 降至 O(N)。

在算法层面，Pareto 前沿与 TOPSIS 的引入使多目标路径规划从概念落地为可运行的模块。Pareto 搜索基于四维代价向量与支配关系剪枝，TOPSIS 通过归一化、加权、理想解距离计算实现科学的折中选择。路径缓存机制在重复查询场景下实现了 95% 以上的命中率。Weidmann 连续密度-速度模型替代了早期的阶梯式函数，消除了速度突变导致的数值振荡。

在工程层面，分层架构的设计与模块化开发策略有效保障了代码的可维护性与可扩展性。仿真引擎的三阶段遍历策略确保了每步内部的一致性；轨迹回放功能将仿真过程中的状态记录与事后可视化拆分为两个独立环节，避免了数据生成与界面渲染的相互耦合。

开发过程中暴露的典型问题——如容量限制过严导致的入口堆积、状态跳转逻辑缺陷导致的死锁——均通过根因分析与系统性修正得以解决。这些经历深化了对"从问题出发选择工具"这一工程思维的理解，也为后续从事复杂系统开发积累了实践经验。
